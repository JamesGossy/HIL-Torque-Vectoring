# Simplify the driver / ECU controllers down to a handful of physics-derived tunables

## Context

The car laps cleanly and fast (fsg2024 ~24.8 s, fse2024 ~18.9 s) but the three
controllers carry **16 runtime tunables** plus ~30 fixed `#define`s. Many are
redundant: four separate numbers all describe "peak lateral grip"
(`g_MAX_LATERAL_ACCEL_MS2`, `g_LAT_GRIP_REF_MS2`, `g_GG_ACCEL_MS2`,
`g_PP_GRIP_ACCEL`), and the steering law is a 7-weight LQR with a Riccati solver
and a linearised bicycle model — far more machinery than this track needs. Several
others (drag feedforward, understeer term, steering limit, hairpin radius floor)
are quantities that are **measurable from `shared/vehicle_config.h`**, so they
should be *computed*, not tuned.

Goal: dramatically cut the tunable/parameter count, deriving from `vehicle_config.h`
wherever a physical formula exists, **without touching the vehicle model or
`vehicle_config.h`**, and keeping pace near current (a balanced cut — accept a
small regression only where it buys real simplicity). Decisions taken with the
user: **balanced aggressiveness**; **all three controllers in scope** to
restructure; **replace the LQR steering law with a Stanley + physics feedforward
law and delete `lqr_steer.c`**.

Target end state: **~5–6 true tunables** (down from 16), one physics-derived
peak-grip function feeding everything, and a steering law with a single gain.

---

## The single grip model (the heart of the simplification)

Add one helper (next to `apex_speed`/`lateral_grip_accel` in
[vehicle_config.h](shared/vehicle_config.h) is forbidden since we can't edit it —
instead put it in a new tiny shared header `shared/grip_model.h`, or as a `static
inline` in each consumer). It expresses the speed-dependent peak lateral grip
straight from physics:

```
peak_lat(v) = MU_TYRE * 9.81 + AERO_GRIP_COEF * v^2        // flat ~10.8 m/s^2
```

`MU_TYRE` (1.1) and `AERO_GRIP_COEF` already exist in `vehicle_config.h`. Every
grip budget then becomes **one dimensionless usage fraction** of this peak:

| Old tunable (value)              | Replacement |
|----------------------------------|-------------|
| `g_MAX_LATERAL_ACCEL_MS2` 12.62  | `GRIP_USE * peak_lat(v)` — the car's corner budget |
| `g_PP_GRIP_ACCEL` 10.0           | `peak_lat(v)` (line shaped for full grip) — no tunable |
| `g_GG_ACCEL_MS2` 8.79            | same as corner budget; drop the separate GG number |
| `g_LAT_GRIP_REF_MS2` 16.5        | `peak_lat(v)` (traction-circle reference) — no tunable |

This collapses **4 grip tunables → 1** (`GRIP_USE`, the fraction of physical grip
the car drives at, ~0.85–0.95). Note the flat peak (~10.8) already exceeds the old
flat `g_MAX_LATERAL_ACCEL_MS2` base, so `GRIP_USE` near 0.9–1.0 reproduces today's
budget; the sweep will land the exact value. The old `g_PLANNER_DOWNFORCE_FRAC`
blend disappears entirely — the planner now just uses `GRIP_USE * peak_lat(v)`,
which is the speed-dependent form the blend was approximating.

## 1. Replace LQR steering with Stanley + feedforward

**Delete** [lqr_steer.c](HIL_Firmware/src/lqr_steer.c) and
[lqr_steer.h](HIL_Firmware/include/lqr_steer.h). Remove the **7 LQR tunables**
(`g_LQR_Q_E1`, `Q_E1D`, `Q_E2`, `Q_E2D`, `R`, `KI`, `I_MAX`).

New steering in [motion_control.c](HIL_Firmware/src/motion_control.c), inline
(replacing the `lqr_steer_command()` call around line 319):

```
// physics-derived understeer gradient, no tuning:
Kus = (MASS_KG / WHEELBASE_M) * (CG_TO_REAR_M/Cf - CG_TO_FRONT_M/Cr)
      with Cf,Cr from the Pacejka slope (same TYRE_B/C/D math lqr_steer.c used,
      computed once)
delta_ff   = (WHEELBASE_M * kappa + Kus * vx*vx * kappa) / ACK_NOMINAL   // FF
delta_head = e2                                                          // align
delta_cte  = atan2(K_STANLEY * (-e1), vx + 1.0)                          // pull in
steer      = delta_ff + delta_head + delta_cte
```

- `Cf`, `Cr`, `Kus`, `ACK_NOMINAL` (0.23) all come from `vehicle_config.h` math
  already present in the old `build_model()` — lift it into a one-time init.
- **One tunable survives: `K_STANLEY`** (cross-track aggressiveness). Everything
  else is derived.
- Keep the existing `MAX_STEER_RAD` saturation and slew-rate clamp (actuator
  limits, fixed). Keep `lqr_steer_reset()`'s role as a no-op or fold its absence
  into `motion_control_reset()` (drop the integrator — Stanley needs none).

Net steering: **7 tunables → 1**, one source file deleted, Riccati loop gone.

## 2. Simplify the speed planner ([motion_control.c](HIL_Firmware/src/motion_control.c))

- `plan_target_speed()` / `brake_decel_avail()`: replace every grip reference with
  the single `GRIP_USE * peak_lat(v)` (corner budget) and `peak_lat(v)` for the
  friction-circle radius. Drop `g_GG_ACCEL_MS2` and the
  `g_PLANNER_DOWNFORCE_FRAC` blend (lines ~113–117 collapse to
  `v_limit = apex_speed(GRIP_USE*flat_peak, kappa, TARGET_SPEED_MS)`).
- Multi-spacing curvature stencil (the `for d=1..3` max): keep — it's cheap and
  not a tunable, just guards against under-reading a sharp apex. Not worth the
  speed risk to remove.
- `MAX_BRAKE_DECEL_MS2` (5.6) and `DRAG_FF_NM` (0.259): **derive**.
  `DRAG_FF_NM = 0.5*AIR_DENSITY*CDA*AERO_AREA*WHEEL_RADIUS_M/GEAR_RATIO` scaling →
  compute the per-(m/s) drag-torque coefficient from aero constants instead of the
  hardcoded 0.259. `MAX_BRAKE_DECEL_MS2` can stay a fixed `#define` (regen limit)
  — it is a powertrain bound, not a grip knob.

## 3. Simplify the path planner ([path_planning.c](HIL_Firmware/src/path_planning.c))

Balanced call — **keep the coordinate-descent lap-time optimiser** (it earns real
lap time via late apexes; a pure min-curvature line measurably regresses on a
corner-heavy track), but strip its tunables:

- `g_PP_GRIP_ACCEL` → use `peak_lat(v)` directly (the line is shaped for full
  physical grip). **Eliminated.**
- `g_PP_MIN_RADIUS_M` (6.03) → **derive** from the steering limit and corner
  budget: `r_min = v_apex^2 / (GRIP_USE*peak_lat)` evaluated at the budget, or more
  simply the kinematic `WHEELBASE_M / tan(MAX_WHEEL_ANGLE)`. Make it a derived
  constant, not a tunable. (Confirm via eval it still opens the hairpin enough.)
- `g_RACING_MARGIN` (0.259): **keep as a tunable** — it's a corridor-safety margin
  tied to tracker error/cone accuracy, genuinely not derivable, and it earns
  clean laps. One tunable.
- `MAX_STEER_RAD` (1.7): derive from `r_min`/Ackermann, or leave fixed (actuator
  limit). Low priority.

Path planner: **3 tunables → 1** (`RACING_MARGIN`).

## 4. Simplify torque vectoring ([torque_vectoring.c](ECU_Firmware/src/torque_vectoring.c))

- **`g_KP_YAW_DEFAULT` (86.2)**: keep ONE master yaw gain. It can be *seeded* from
  `YAW_INERTIA_KGM2` (a natural yaw-moment scale is `Iz/something`), but the exact
  value still wants tuning — keep it a tunable but it's the only TV gain that must
  be.
- **`g_TV_KFF` (10.4)**: the feedforward bias `KFF*desired_yaw_rate*v`. Fold the
  feedforward into the same physical basis as Kp so it is **one derived ratio of
  Kp**, not an independent tunable (like `TV_KI_FRAC`/`TV_KD_FRAC` already are).
  Eliminates 1 tunable.
- `TV_K_US` (0.06): **derive** from the same understeer gradient `Kus` computed for
  steering (§1) — it is literally the understeer term in the yaw reference. Drop
  the magic 0.06. **Eliminated as a knob.**
- `TV_SPEED_REF_MS` (12), `TV_WHEEL_YAW_TRUST` (0.25), `TV_YAW_DEADBAND` (0.03):
  fixed `#define`s already. Consider dropping the wheel-speed/IMU fusion entirely
  (use IMU yaw only) to remove `TV_WHEEL_YAW_TRUST` and the whole Step-1b block —
  the IMU is the trustworthy source and this simplifies ~10 lines. Verify no
  regression in eval; if it costs cleanliness, keep fusion.
- `TV_KI_FRAC`/`TV_KD_FRAC`/`TV_I_MAX_NM`: keep as fixed fractions (not tunables
  today).
- Grip-aware bleed distributor (`grip_limits()` + Steps 6–8): **keep** — it's the
  physics-derived per-corner allocation and uses only `vehicle_config.h` constants,
  exactly the style the user wants. No tunables in it.

TV: **2 tunables → 1** (`KP_YAW`), plus optional removal of the fusion `#define`.

## Final tunable set (target)

| Tunable | Role | Why it survives |
|---------|------|-----------------|
| `GRIP_USE` | fraction of physical peak grip the car drives at | the one real grip knob; everything else derives from it |
| `K_STANLEY` | steering cross-track aggressiveness | only steering gain after LQR removal |
| `RACING_MARGIN` | racing-line corridor safety margin | tied to tracker/cone error, not physics |
| `KP_YAW` | master TV yaw gain | yaw authority; FF/Ki/Kd are fractions of it |

That's **4 tunables, down from 16** (optionally a 5th if Stanley wants a small
heading-vs-cross-track balance, or `GRIP_USE` is split planner/line). Everything
else becomes a derived constant or a fixed actuator/powertrain `#define`.

1. `make format-check` then build clean (Windows temp caveat: invoke gcc directly
   with `TMPDIR/TMP/TEMP=/c/msys64/tmp` if `make` can't set temp).
2. `make test` — unit tests (rewrite the LQR test for Stanley).
3. **Re-tune the 4 survivors** with `tools/tool_smart_sweep_lqr_multi.py` (the
   shared two-track set), then paste winners into `tunables.c`.
4. `make eval` (default fsg2024) **and** `TRACK=fse2024 make eval`: require
   **0 off-track ticks** and **a completed lap** on both. Compare mean/worst CTE
   and lap time to the baseline (fsg2024 ~24.8 s, fse2024 ~18.9 s) — accept a small
   regression (target within ~1 s/lap) as the agreed cost of the cut; investigate
   anything larger.
5. `tools/tool_robust_check_lqr.py` on the final config — must survive ±3%
   perturbation with 0 off-track (knife-edge configs are invalid per repo policy).
